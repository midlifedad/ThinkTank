<objective>
Build the content attribution handler (`tag_content_thinkers`) and trigram similarity module for candidate dedup. The attribution handler creates ContentThinker junction rows linking content to thinkers with roles and confidence scores. The trigram module provides pg_trgm-based name deduplication to prevent near-duplicate candidates from accumulating. Write integration tests and contract tests for all Phase 3 handlers.

Purpose: Content attribution is what makes the corpus useful -- without it, content is just rows without connections to thinkers. Trigram dedup ensures the candidate pipeline doesn't accumulate duplicates like "John Smith" and "Dr. John Smith" as separate entries.

Output: 1 trigram query module, 1 handler, updated registry, 2 integration test files, 1 contract test file.
</objective>

    1. Create `src/thinktank/ingestion/trigram.py` with two async functions using raw SQL via `text()` (ORM cannot express pg_trgm operations):

    `async def find_similar_candidates(session: AsyncSession, normalized_name: str, threshold: float = 0.7) -> list[tuple[str, str, float]]`:
    Query `candidate_thinkers` for rows where `similarity(normalized_name, :name) > :threshold`. Return list of (id, name, similarity_score). Order by similarity descending.

    `async def find_similar_thinkers(session: AsyncSession, normalized_name: str, threshold: float = 0.7) -> list[tuple[str, str, float]]`:
    Query `thinkers` for rows where `similarity(lower(name), :name) > :threshold`. Return list of (id, name, similarity_score). Prevents candidates that already exist as thinkers.

    Use the SQL from RESEARCH.md code examples. Cast UUIDs to str in the return values.

    2. Create `src/thinktank/handlers/tag_content_thinkers.py` with `async def handle_tag_content_thinkers(session: AsyncSession, job: Job) -> None`.

    Handler logic:

    a. Extract `content_ids`, `source_id`, and `descriptions` from `job.payload`. content_ids is a list of content ID strings. source_id is a string. descriptions is a dict mapping content_id (str) to description text (str). This is the canonical payload schema -- fetch_podcast_feed (Plan 03) enqueues this exact structure.
    b. Load all active approved thinkers: `select(Thinker).where(Thinker.active == True, Thinker.approval_status == "approved")`. Build thinker_names list: `[{"id": t.id, "name": t.name} for t in thinkers]`.
    c. Load the source to get source_owner: `source = await session.get(Source, UUID(source_id))`. Get source owner thinker name.
    d. For each content_id in content_ids:
       - Load content: `content = await session.get(Content, UUID(content_id))`
       - If content is None or content.status == 'skipped', skip (don't attribute skipped content).
       - Get the episode description from the payload: `description = descriptions.get(content_id, "")`.
       - Call `match_thinkers_in_text(content.title, description, thinker_names, source_owner_name)` to get attribution matches.
       - For each match result, create a ContentThinker row: `session.add(ContentThinker(content_id=content.id, thinker_id=match["thinker_id"], role=match["role"], confidence=match["confidence"]))`.
       - Check for duplicate ContentThinker before insert (composite PK on content_id + thinker_id). Use `session.get(ContentThinker, (content.id, match["thinker_id"]))` or handle IntegrityError.
       - For names found in text that do NOT match any existing thinker, check if they should become candidates. This is a precursor to cascade discovery (Phase 6). For now, extract unmatched names and:
         * Normalize the name via `normalize_name()`
         * Check `find_similar_thinkers()` -- if match above 0.7, skip (already a thinker under different name)
         * Check `find_similar_candidates()` -- if match above 0.7, increment `appearance_count` and update `last_seen_at` on existing candidate
         * If no match in either, create new CandidateThinker with `normalized_name`, `appearance_count=1`, `status='pending_llm'`, `sample_urls=[content.url]`
    e. Commit all changes.
    f. Log: count of attributions created, count of candidates created/updated.

    IMPORTANT SIMPLIFICATION for v1: The name extraction from descriptions for candidate discovery is complex and involves NLP. For this phase, limit candidate creation to names that were EXPLICITLY matched against the thinker list but didn't match. Do NOT attempt general NER/name extraction from descriptions -- that's Phase 6 (DISC-01 scan_for_candidates). The tag_content_thinkers handler in Phase 3 should:
    - Create ContentThinker rows for matches against existing thinkers (the attribution part -- DISC-03)
    - Use trigram dedup when creating candidates from unmatched names found during matching (the dedup part -- DISC-04)
    - But NOT scan for arbitrary new names in text -- that's cascade discovery (Phase 6)
