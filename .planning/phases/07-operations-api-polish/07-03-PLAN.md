    1. Create `docs/operations-runbook.md` with the following sections:

       **1. Bootstrap (Fresh Deployment)**
       - Prerequisites: PostgreSQL running, DATABASE_URL set, alembic upgrade head completed
       - Run: `python -m scripts.bootstrap` (or `uv run python -m scripts.bootstrap`)
       - Expected output: category count, config count, thinker count, workers_active=true
       - Verification: Check admin dashboard at :8001/admin/, verify queue depth shows pending llm_approval_check jobs

       **2. Post-Deploy Verification**
       - Health check: `curl http://localhost:8000/health` returns `{"status": "healthy"}`
       - Admin dashboard: `curl http://localhost:8001/admin/` returns 200
       - API docs: `curl http://localhost:8000/docs` returns 200
       - Job queue: `curl http://localhost:8000/api/jobs/status` shows job counts
       - Worker status: Check system_config `workers_active` is true

       **3. Rollback**
       - Stop workers: `PUT /api/config/workers_active {"value": false, "set_by": "admin"}`
       - Revert migration: `alembic downgrade -1`
       - Redeploy previous version
       - Re-run migration: `alembic upgrade head`
       - Restart workers

       **4. Common Problems**
       - Workers not processing jobs: Check workers_active config, check kill switch, check error log
       - LLM reviews timing out: Check Anthropic API key, check rate limits, check llm_timeout_hours config
       - GPU not scaling: Check Railway API key, check gpu_service_status in system_config, check rate limits
       - Feed polling failures: Check source error_count, check RSS feed URL accessibility, check rate limits
       - High error rate: Check admin dashboard error log, check error categories, review job payloads
       - Database connection issues: Check DATABASE_URL, check connection pool (pool_size, max_overflow), check PostgreSQL logs

       **5. Operational Commands**
       - Pause system: `PUT /api/config/workers_active {"value": false, "set_by": "admin"}`
       - Resume system: `PUT /api/config/workers_active {"value": true, "set_by": "admin"}`
       - Check costs: Admin dashboard cost tracker or `GET /api/config/` for rate limits
       - Manual LLM override: Admin dashboard LLM Panel -> Override button
       - Re-run bootstrap seeds: `python -m scripts.seed_categories` (idempotent)

    2. Create `docs/development-guide.md` with the following sections:

       **1. Project Structure**
       - Overview of src/thinktank/ directory layout
       - Separation of concerns: api/ (REST), admin/ (dashboard), handlers/ (job logic), models/ (ORM), queue/ (job queue), worker/ (loop)

       **2. Adding a New Job Type**
       - Step 1: Create handler file in `src/thinktank/handlers/` following JobHandler protocol
       - Step 2: Register in `src/thinktank/handlers/registry.py`
       - Step 3: Create contract test in `tests/contract/`
       - Step 4: Add error category extensions if needed in `src/thinktank/queue/errors.py`
       - Example: Walk through creating a hypothetical `cleanup_old_content` handler

       **3. Adding a New API Endpoint**
       - Step 1: Add Pydantic schemas in `src/thinktank/api/schemas.py`
       - Step 2: Create router in `src/thinktank/api/routers/`
       - Step 3: Include router in `src/thinktank/api/main.py`
       - Step 4: Write contract tests in `tests/contract/`
       - Conventions: use get_session dependency, SQLAlchemy 2.0 select(), from_attributes=True

       **4. Adding a New Thinker Category**
       - Option A: Via admin dashboard category management
       - Option B: Add to TAXONOMY dict in `scripts/seed_categories.py` and re-run
       - Category naming conventions: lowercase slugs, descriptive names

       **5. Testing Conventions**
       - Test pyramid: unit (pure logic), integration (real Postgres), contract (endpoint shapes)
       - Factory functions in `tests/factories.py`
       - Fixtures: session, client, _cleanup_tables
       - Pattern: `pytestmark = pytest.mark.anyio` on all async test files
       - Running tests: `uv run pytest tests/unit -x -q` (quick), `uv run pytest tests/ -x` (full)

       **6. Database Conventions**
       - SQLAlchemy 2.0 style: `select()` + `session.execute()`
       - Timezone-naive datetimes: `datetime.now(UTC).replace(tzinfo=None)`
       - LOCALTIMESTAMP for raw SQL comparisons
       - JSONB for flexible data (system_config, payloads)
       - Advisory locks for migration safety

       **7. Deployment**
       - 4 Railway services: API (port 8000), Admin (port 8001), CPU Worker, GPU Worker
       - Docker files: Dockerfile.api, Dockerfile.admin, Dockerfile.worker-cpu, Dockerfile.worker-gpu
       - Environment variables: DATABASE_URL, ANTHROPIC_API_KEY, RAILWAY_API_KEY
