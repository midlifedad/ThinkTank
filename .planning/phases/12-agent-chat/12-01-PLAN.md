Create new package `src/thinktank/agent/` with 4 modules:

**1. `system_prompt.py`** -- `build_chat_system_prompt() -> str`
Build a system prompt that tells the agent:
- It is the ThinkTank assistant for a content ingestion pipeline admin panel
- It has access to two tools: `query_database` and `propose_action`
- The database schema summary (table names, key columns, relationships) -- hardcode this as a string constant derived from the models. Include: thinkers, sources, content, jobs, candidate_thinkers, llm_reviews, system_config, categories, content_thinkers, rate_limit_usage, api_usage
- For `query_database`: generates read-only SQL (SELECT only), the tool executes it and returns results as JSON
- For `propose_action`: creates a structured proposal with action_type (one of: "add_thinker", "approve_source", "reject_source", "trigger_discovery", "toggle_kill_switch", "update_config", "retry_job", "cancel_job"), target (what it acts on), and details (dict with action parameters). The proposal is shown to the user who must click Confirm
- Safety rules: NEVER generate INSERT/UPDATE/DELETE SQL. All mutations go through propose_action. Keep SQL queries bounded (LIMIT 50 default). Do not access secrets or API keys
- Tone: concise, direct, uses data from queries to support answers

**2. `tools.py`** -- Tool definitions and execution
- `AGENT_TOOLS`: list of dicts matching Anthropic tool_use format with name and input_schema (JSON Schema)
  - `query_database`: input_schema has `sql` (string, the SELECT query) and `explanation` (string, what this query answers)
  - `propose_action`: input_schema has `action_type` (enum string), `target` (string description), `details` (object with action params), `explanation` (string, why this action)
- `async def execute_tool(tool_name: str, tool_input: dict, session: AsyncSession) -> dict`:
  - For `query_database`: validate SQL starts with SELECT (case-insensitive, stripped). Execute via `session.execute(text(sql))`. Return `{"rows": [...], "row_count": N}` where each row is a dict. Catch exceptions and return `{"error": str(e)}`. Add LIMIT 50 if no LIMIT clause present
  - For `propose_action`: do NOT execute anything. Return `{"proposal": {"action_type": ..., "target": ..., "details": ..., "explanation": ...}, "status": "awaiting_confirmation"}`. The frontend will render a Confirm button
  - For unknown tools: return `{"error": "Unknown tool"}`
- `async def execute_confirmed_action(action_type: str, details: dict, session: AsyncSession) -> dict`:
  - This runs AFTER user clicks Confirm. Implement each action_type:
    - `add_thinker`: INSERT into thinkers (name from details, generate slug, tier=3, bio="Added via chat agent", approval_status="awaiting_llm"), then create an llm_approval_check job. Return `{"success": True, "message": "..."}`
    - `approve_source`: UPDATE sources SET approval_status='approved' WHERE id=details["source_id"]. Create LLMReview with trigger=admin_override. Return success
    - `reject_source`: UPDATE sources SET approval_status='rejected' WHERE id=details["source_id"]. Create LLMReview with trigger=admin_override. Return success
    - `trigger_discovery`: INSERT into jobs (job_type='discover_guests_podcastindex', payload={"thinker_id": details["thinker_id"]}). Return success
    - `toggle_kill_switch`: Update system_config workers_active (flip boolean). Return new state
    - `update_config`: Update system_config row by key. Return success
    - `retry_job`: Create new pending job copying job_type and payload from the failed job id. Return success
    - `cancel_job`: UPDATE jobs SET status='cancelled' WHERE id=details["job_id"] AND status='pending'. Return success or error if not pending
  - Wrap all in try/except, return `{"error": str(e)}` on failure. Commit after successful mutation

**3. `session.py`** -- In-memory chat session management
- `ChatMessage` dataclass: role (str: "user"|"assistant"|"tool_use"|"tool_result"), content (str), tool_name (str|None), tool_input (dict|None), tool_use_id (str|None), timestamp (datetime)
- `ChatSession` dataclass: session_id (str), messages (list[ChatMessage]), pending_proposals (dict[str, dict] -- keyed by proposal_id), created_at (datetime)
- `ChatSessionStore` class:
  - `_sessions: dict[str, ChatSession]` (in-memory)
  - `create() -> ChatSession`: generates UUID session_id, returns new session
  - `get(session_id: str) -> ChatSession | None`
  - `add_message(session_id, message: ChatMessage)`: appends to session messages
  - `add_proposal(session_id, proposal_id, proposal_data)`: stores pending proposal
  - `pop_proposal(session_id, proposal_id) -> dict | None`: removes and returns proposal
  - `get_anthropic_messages(session_id) -> list[dict]`: converts message history to Anthropic API format (list of {"role": "user"|"assistant", "content": ...} dicts). Tool use and tool results must be formatted per Anthropic API spec: assistant message with tool_use content block, followed by user message with tool_result content block
  - `cleanup_old(max_age_hours=4)`: remove sessions older than max_age_hours
- Create a module-level singleton: `chat_sessions = ChatSessionStore()`

**4. `stream.py`** -- Streaming agent response via Anthropic API
- `async def stream_agent_response(session_id: str, user_message: str, db_session: AsyncSession) -> AsyncGenerator[str, None]`:
  - Get or create ChatSession from chat_sessions store
  - Add user message to session
  - Build messages list via `chat_sessions.get_anthropic_messages(session_id)`
  - Get Anthropic client via `get_secret(db_session, "anthropic_api_key")`
  - Use `AsyncAnthropic.messages.stream()` context manager (NOT .create()) with:
    - model="claude-sonnet-4-20250514"
    - system=build_chat_system_prompt()
    - messages=messages_list
    - tools=AGENT_TOOLS
    - max_tokens=4096
  - Iterate over stream events. Yield SSE-formatted strings:
    - On text delta: yield `data: {"type": "text_delta", "text": "..."}\n\n`
    - On tool_use block complete (input_json accumulated): execute the tool via execute_tool(), then:
      - If tool is propose_action: generate a proposal_id (uuid4), store in session via add_proposal, yield `data: {"type": "proposal", "proposal_id": "...", "action_type": "...", "target": "...", "details": {...}, "explanation": "..."}\n\n`
      - If tool is query_database: yield `data: {"type": "tool_result", "tool": "query_database", "result": {...}}\n\n`, then continue the conversation by calling the API again with the tool result appended (so the agent can interpret the query results for the user). For this continuation, stream text deltas the same way
    - On message end: yield `data: {"type": "done"}\n\n`
  - After streaming completes, add the full assistant message (text + any tool_use blocks) to the session
  - Handle the tool-use loop: if the response stop_reason is "tool_use", execute the tool, add tool_result to messages, and make another streaming API call. Repeat until stop_reason is "end_turn" or max 5 iterations (to prevent infinite loops)
  - On any exception: yield `data: {"type": "error", "message": "..."}\n\n`

**1. `src/thinktank/admin/routers/chat.py`** -- Chat API endpoints

Create router with `prefix="/admin/chat"`, `tags=["chat"]`.

Three endpoints:

**POST `/admin/chat/send`** -- Send a message and stream the response
- Accept form data: `message` (str), `session_id` (str, optional -- if empty, create new session)
- Import `EventSourceResponse` from `sse_starlette.sse`
- If no session_id provided, create a new session via `chat_sessions.create()`, include the new session_id in the first SSE event: `data: {"type": "session_init", "session_id": "..."}\n\n`
- Return `EventSourceResponse(stream_agent_response(session_id, message, db_session))` with `media_type="text/event-stream"`
- The db_session should be obtained via a dependency but kept open for the duration of streaming. Use `async_session_factory()` directly (not the Depends generator) since SSE responses outlive the request lifecycle. Create session in the generator passed to EventSourceResponse, close it after iteration completes

**POST `/admin/chat/confirm/{proposal_id}`** -- Execute a confirmed action
- Accept path param `proposal_id` (str) and form data `session_id` (str)
- Pop the proposal from `chat_sessions.pop_proposal(session_id, proposal_id)`
- If proposal is None: return 404 JSON `{"error": "Proposal not found or already executed"}`
- Call `execute_confirmed_action(proposal["action_type"], proposal["details"], db_session)`
- Return JSON result (the dict from execute_confirmed_action)
- Use `Depends(get_session)` since this is a normal request-response cycle

**GET `/admin/chat/history`** -- Get chat history for rendering
- Accept query param `session_id` (str)
- Get session from `chat_sessions.get(session_id)`
- If None: return 404
- Return JSON list of messages: `[{"role": ..., "content": ..., "timestamp": ..., "proposal": ... (if tool_name=="propose_action")}]`
- Filter out tool_use and tool_result messages from the response -- only return user and assistant text messages, plus proposals
